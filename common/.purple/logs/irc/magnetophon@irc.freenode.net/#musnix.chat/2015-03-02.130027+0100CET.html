(02:32:34 PM) magnetophon: goibhniu: OK, full disclosure, there is still some work to be done to make VOF really user friendly, (iow for a smooth demo): faust2jack creates a nice gui, and can receive osc data, but it has a crude way of handling 'presets': it creates .$PROGRAMNAMErc files on shutdown so that when you start it again it loads the sound you had on shutdown. If there is no .rc file it uses the default setting you define in the source code. The solutions in order of increasing workload+payoff are: 1) Use just the default sounds in the demo. This is not as bad as it seems, since I need to split up my monolithic beast in modules anyway, to reduce the cpu-load. (so you'd have a string synth, a bass synth, and a lead, for example) 2) symlink or copy .rc files into place before startup. 3) feed it parameter data over OSC via an external program (pd or supercollider for example). This last one is coolest, since you can automate parameter changes at runtime, as part of the demo song.
